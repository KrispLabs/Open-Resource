# Open Resource — Deployment Guide

Production deployment of the Open Resource hiring platform.

- **Frontends** (`apps/hr`, `apps/applicant`, `apps/dev`) → **Vercel** (static Vite SPAs)
- **Backend** (`apps/backend`) → **Render** (Docker web service, free tier)

> **Why not the backend on Vercel/serverless?** The backend uses Server-Sent Events
> (live scoring stream), long-running `asyncio` background tasks (batch scoring,
> outbound campaigns), a SQLite file, and local PDF uploads. Serverless functions
> buffer responses, time out, and have an ephemeral filesystem — they cannot run this
> backend correctly. It needs a persistent, always-on runtime. See `CLAUDE.md` →
> *Key Constraints*.

---

## Architecture

```
  ┌─────────────────────── Vercel (static CDN) ───────────────────────┐
  │  hr.<vercel>        applicant.<vercel>        dev.<vercel>         │
  │  apps/hr/dist       apps/applicant/dist       apps/dev/dist        │
  └───────────────┬───────────────┬───────────────┬───────────────────┘
                  │  VITE_API_URL (https, set at build time)
                  ▼
  ┌──────────────────────── Render (Docker, always-on) ───────────────┐
  │  open-resource-api.onrender.com                                    │
  │  FastAPI + Uvicorn   ·   SSE   ·   SQLite hireai.db   ·   uploads/  │
  └────────────────────────────────────────────────────────────────────┘
```

The frontend API client (`packages/shared/apiClient.ts`) uses `VITE_API_URL` as the
**bare backend origin** (no trailing `/api`). Backend routers self-prefix
(`/auth`, `/jobs`, `/api/dev`, …), so the value is e.g.
`https://open-resource-api.onrender.com`.

## Free-tier caveats (read before demoing)

Render's free web service:

1. **Spins down after ~15 min idle** → the next request cold-starts (~50 s). While you
   are actively using the app it stays warm, so SSE and background tasks behave normally.
2. **No persistent disk** → `hireai.db` and uploaded resumes are wiped on spin-down or
   redeploy. The DB auto-recreates and re-seeds demo accounts on every boot, so the app
   always *works* — data just isn't durable.

**Free path to durable data (optional):** create a free [Neon](https://neon.tech)
Postgres database and set `DATABASE_URL=postgresql+psycopg://USER:PASS@HOST/DB` on Render.
SQLAlchemy works with Postgres unchanged. (Uploaded files would still be ephemeral on the
free tier — move `UPLOAD_DIR` to object storage or a paid Render disk for full durability.)

## Prerequisites

- A [Vercel](https://vercel.com) account + `vercel` CLI (`npm i -g vercel`) or a Vercel API token.
- A [Render](https://render.com) account (free).
- The repo pushed to GitHub: `github.com/KrispLabs/Open-Resource` (branch `outbound-v3-rebuild`).
- Provider API keys: Featherless AI, GitHub PAT, (optional) Bright Data.

## Step 1 — Deploy the backend to Render

The backend must go first so the frontends know its URL.

**Option A — Blueprint (recommended):**

1. Push `render.yaml` (already in repo root) to GitHub.
2. Render dashboard → **New → Blueprint** → connect `KrispLabs/Open-Resource` → select `render.yaml`.
3. Render prompts for the `sync: false` env vars. Set:

   | Variable | Value |
   |----------|-------|
   | `FEATHERLESSAI_API_KEY` | your Featherless key |
   | `GITHUB_TOKEN` | your GitHub PAT (read-only is fine) |
   | `BRIGHTDATA_API_KEY` | your Bright Data key (or leave blank → GitHub REST fallback) |
   | `HR_PASSWORD` | `demo1234` (or your own) |
   | `DEV_PASSWORD` | `demo1234` (or your own) |
   | `FRONTEND_ORIGINS` | leave blank for now — set in Step 3 |

   `JWT_SECRET` and `SERVER_SECRET_KEY` are auto-generated by Render.
4. Click **Apply**. First build takes a few minutes (Docker image).
5. Verify: `curl https://open-resource-api.onrender.com/health` → `{"status":"ok",...}`.

**Option B — Manual:** New → Web Service → connect repo → Runtime *Docker* →
Root Directory `apps/backend` → Plan *Free* → Health Check Path `/health` → add the
env vars from `render.yaml` manually.

> **Note your backend URL** (e.g. `https://open-resource-api.onrender.com`) — needed next.

---

## Step 2 — Deploy the frontends to Vercel

Each app is a separate Vercel project with its **Root Directory** set to the app folder.
Vercel auto-detects the pnpm workspace and installs from the repo root.

For **each** app (`hr`, `applicant`, `dev`):

```bash
cd apps/<app>
vercel link            # create/link a project; set Root Directory = apps/<app>
# Set the backend URL as a BUILD-TIME env var (Vite inlines import.meta.env):
vercel env add VITE_API_URL production   # paste your Render URL
vercel --prod          # build + deploy
```

`apps/<app>/vercel.json` already configures the Vite framework preset, `dist` output, and
the SPA rewrite (`/(.*) → /index.html`) so client-side routes (React Router) work on refresh.

Record the three production URLs Vercel returns, e.g.:
- `https://open-resource-hr.vercel.app`
- `https://open-resource-applicant.vercel.app`
- `https://open-resource-dev.vercel.app`

---

## Step 3 — Wire CORS (close the loop)

Set the backend's allowed origins to the three Vercel production URLs:

- Render dashboard → `open-resource-api` → **Environment** →
  `FRONTEND_ORIGINS = https://open-resource-hr.vercel.app,https://open-resource-applicant.vercel.app,https://open-resource-dev.vercel.app`
- Save → Render redeploys automatically.

`*.vercel.app` **preview** URLs are already allowed via `cors_origin_regex` in
`config.py`, so preview deploys work without extra config. Add any custom domains to
`FRONTEND_ORIGINS` as well.

## Environment variables reference

### Backend (Render)

| Variable | Required | Notes |
|----------|----------|-------|
| `APP_ENV` | yes | `production` — enables the insecure-default secret guard |
| `JWT_SECRET` | yes | strong random; auto-generated by Render |
| `SERVER_SECRET_KEY` | yes | AES master key (SHA-256'd to 32 bytes); auto-generated |
| `JWT_EXPIRE_DAYS` | no | default `7` |
| `DATABASE_URL` | yes | `sqlite:///./hireai.db` (or a Neon Postgres URL) |
| `UPLOAD_DIR` | yes | `uploads` |
| `HR_EMAIL` / `HR_PASSWORD` | yes | seeded HR account |
| `DEV_EMAIL` / `DEV_PASSWORD` | yes | seeded dev/admin account |
| `FEATHERLESSAI_API_KEY` | yes* | AI scoring / JD analysis / outreach |
| `GITHUB_TOKEN` | yes* | GitHub sourcing (5000 req/hr) |
| `BRIGHTDATA_API_KEY` | no | optional; falls back to GitHub REST if empty |
| `BRIGHTDATA_DATASET_ID` | no | default in `config.py` |
| `BRIGHTDATA_SERP_DATASET_ID` | no | default in `config.py` |
| `FRONTEND_ORIGINS` | yes | comma-separated Vercel/custom origins |

\* App boots without these (startup only logs an error), but the corresponding AI/sourcing
features fail until set.

### Frontend (Vercel) — set per project, **Production** + **Preview**

| Variable | Value |
|----------|-------|
| `VITE_API_URL` | your Render backend origin, e.g. `https://open-resource-api.onrender.com` |

## Troubleshooting

| Symptom | Cause / fix |
|---------|-------------|
| Frontend loads, every API call CORS-fails | `FRONTEND_ORIGINS` on Render missing the exact Vercel origin (scheme + host, no trailing slash). |
| First request after idle takes ~50 s | Render free-tier cold start. Expected; hit `/health` to warm it. |
| Data disappeared / resumes 404 | Free-tier ephemeral disk reset. Use Neon Postgres + object storage for durability. |
| 500 on AI features | Provider key missing/invalid. Check Render logs and `/health` (`featherless`, `github_sourcing`). |
| Backend won't boot, `FATAL: ... insecure default` | `APP_ENV=production` but `JWT_SECRET`/`SERVER_SECRET_KEY` not set. Let Render generate them. |
| 404 on deep links / refresh (e.g. `/jobs/123`) | SPA rewrite missing — ensure `apps/<app>/vercel.json` is deployed. |
| pnpm install fails on Vercel | Ensure project Root Directory = `apps/<app>` and "Include source files outside Root Directory" is on (default for workspace projects). |
